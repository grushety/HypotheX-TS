## UI-011 Compensation-mode selector (naive / local / coupled)

Created `frontend/src/lib/constraints/createCompensationModeSelectorState.js` (pure state) and `frontend/src/components/constraints/CompensationModeSelector.vue` (segmented control).  State exports `COMPENSATION_MODES = ['naive','local','coupled']`, `MODE_TOOLTIPS` (AC-spec strings), `MODE_LABELS`, `defaultModeForDomain(domainHint)` (mirrors OP-051 backend table — hydrology→local, seismo-geodesy→coupled, remote-sensing→local, other→naive; case-insensitive), `isCompensationRequired(domainHint, opCategory)` (true iff domain ∈ {hydrology, seismo-geodesy} AND op_category ∈ {plateau, trend, step, transient}), `nextMode(current, dir)` for keyboard cycling, and `createCompensationModeSelectorState({...})` returning `{mode, defaultMode, isRequired, hasExplicitChoice, canSubmit, choices: [...{mode, label, tooltip, isSelected, isRecommended}]}`.  **Component is an ARIA radiogroup** with `role="radiogroup"` + per-button `role="radio"` + `aria-checked`; roving tab-index (only the selected button has `tabindex=0`); keyboard ArrowLeft/Up = previous, ArrowRight/Down = next, Home = first, End = last with WAI-ARIA "selection follows focus" semantics; per-button `:title` carries the AC tooltip; "★" recommended badge on the domain-default mode when it's *not* the currently selected one (so user sees both their override and the recommendation simultaneously).  Required-mode visual: red asterisk in label + red border on the group + red hint paragraph "Choose a compensation mode to confirm this op." when `isRequired && !hasExplicitChoice`.  **Critical bug caught in code review** (option-a fix applied): my first version had `watch(() => state.value.mode, ..., { immediate: true })` to surface the resolved default to a v-model parent if `props.modelValue == null` on mount.  This silently emitted the default → parent's modelValue became valid → `validSelected = true` → `explicit = true` → `canSubmit = true`, defeating the required-mode gate without any user interaction.  Fix: removed the immediate-emit watch entirely; parents that want a pre-fill must read `defaultModeForDomain()` themselves.  Documented inline with a comment + a contract warning that passing a non-null `modelValue` without `hasExplicitChoice: true` will silently satisfy the gate (the state-module conflates "model has a value" with "explicit"), so parents binding v-model to a field they pre-initialise for *display* must keep that field `null` until the user clicks OR pass `hasExplicitChoice: false`.  **Op-card integration is out of scope for this ticket** — the component emits `update:modelValue` and `change`; the parent (OP-020..026 / OP-032 op cards) is responsible for forwarding the selection into the OP-041 chip.  38 tests in `createCompensationModeSelectorState.test.js`; npm test glob already covered `lib/constraints/*.test.js`; 343 frontend tests pass; `npm run build` clean (132.96 kB JS / 21.47 kB CSS gzipped).  Code-reviewer initially flagged the auto-emit bug as BLOCKING; after the fix the re-review approved with a non-blocking nit about the contract warning that was applied in the same commit.

## UI-010 Constraint-budget bar (per-law residual + per-component breakdown)

Created `frontend/src/lib/constraints/createConstraintBudgetState.js` (pure state) and two Vue components: `ConstraintBudgetBar.vue` + `ConstraintResidualBreakdown.vue`.  The state module exports `classifyResidual`, `formatResidual`, `classifyDirection`, `createConstraintBudgetState`, `buildBreakdown` plus constants `HARD_LAWS = {phase_closure, nnr_frame}`, `SOFT_LAWS = {water_balance, moment_balance}`, `DEFAULT_TOLERANCE` (mirrors OP-032 backend), `STATUS = {GREEN, AMBER, RED}`, `FILL_CAP = 1.5`, `SUPPORTED_BREAKDOWN_LAWS`.  **Status logic**: `|r| ≤ tolerance` → green; over tolerance + hard law → red; over tolerance + soft (or unknown) law → amber.  **Residual formatting** (`formatResidual`): scientific notation when `|r| < 0.001`, otherwise fixed precision with units; `null` and `NaN` render as em-dash; sign preserved so users see over-budget vs under-budget direction at a glance.  **`createConstraintBudgetState` returns 18 fields** including `fillFraction` (capped at `FILL_CAP=1.5` so a runaway 10⁶ × tolerance residual doesn't break the UI), `initialFillFraction`, `showPrePost` (true only when `compensationMode != 'naive'` *and* both initial+final residuals are present), `direction` (improving / worsening / unchanged), `ariaText` (richer for screen readers — includes law name, status word, direction, both residuals when showPrePost), and `hoverText` matching the AC format `Δ = X (of Y tolerance)`.  **`buildBreakdown`** decodes `constraint_residual` components into per-component contribution rows using sign tables that mirror each law's equation: water `+P, −ET, −Q, −ΔS`; moment `+Mxx, +Myy, +Mzz`; phase `+φ12, +φ23, −φ13`; NNR `+ωx, +ωy, +ωz`.  Skips non-finite components defensively; returns `supported=false` for unknown laws.  **`ConstraintBudgetBar.vue`** renders a horizontal CSS bar (no SVG) with `currentColor` fill + status-class background, click-to-expand toggle on a `<button>` (Enter/Space-keyboard accessible, focus-visible outline), `aria-live="polite"` region with the full residual announcement (visually-hidden via the standard sr-only `clip` pattern — the visible `.constraint-budget-bar__readout` span shows the same numeric for sighted users), and a `:title` attribute carrying the AC-spec hover text.  **CSS gotcha**: the bar's tolerance marker is at `66.6667%` because `1/FILL_CAP = 1/1.5`; the JS template's fill-style math `width: ${fillFraction * (100/1.5)}%` mirrors this — both depend on `FILL_CAP=1.5` and would both need updating if it changes (documented inline).  **Controlled-prop watcher**: the `expanded` prop is mirrored into a local `ref` AND `watch`-ed so that a parent passing `expanded` reactively (controlled mode) sees updates after mount — without the `watch`, the parent's later changes would silently no-op (code-reviewer caught the trap).  53 tests in `createConstraintBudgetState.test.js`; npm test glob already covered `lib/constraints/*.test.js` so no glob update needed.  358 frontend tests pass; `npm run build` clean (132.96 kB JS, 21.47 kB CSS gzipped 45.56 + 4.48).  Code-reviewer approved with no blocking issues; flagged `66.6667%` magic number as worth a future `--fill-cap` CSS custom property when convenient.

## OP-051 Compensation-mode projection (naive / local / coupled)

Created `backend/app/services/operations/tier3/compensation.py` — the projection primitive that OP-032 / OP-050 / cf_coordinator have been deferring to with a `try/except ImportError` fallback to a no-op `_naive_project`.  Public surface: `project(X_edit, constraint, compensation_mode='local', *, segment_mask=None, tolerance=1e-6, max_iter=20) → np.ndarray`, `default_compensation_mode_for_domain(domain_hint)`, `CompensationMode` literal, `HasJacobian` `@runtime_checkable` Protocol.  **Three modes**: `naive` (residual reported via `logger.info`, X returned unchanged but as a fresh array — never aliases the input), `local` (only values where `segment_mask=True` change; outside-mask values byte-identical), `coupled` (Newton-style iterative projection minimising `½‖X' − X‖²` s.t. `g(X')=0`).  **The compensation-mode selector is the atomic novelty** of HypotheX-TS's conservation-enforcement claim — no existing CF-XAI tool exposes naive/local/coupled as a user choice.  **Critical compatibility move**: the OP-050 `Constraint` Protocol in `cf_coordinator.py:56–71` exposes only `name` / `residual` / `satisfied` — *no* `jacobian`.  My `_jacobian_matrix` checks for `hasattr(constraint, 'jacobian')` first; if absent or raising, falls back to **central differences** (`_numerical_jacobian`, 2n residual evaluations per Newton step, eps=1e-7).  This keeps cf_coordinator's existing `MockConstraint`-based tests passing without modification.  All 37 cf_coordinator tests still pass even though `_get_default_projector` now resolves to the real OP-051 projector — they all override `projector=Mock`.  **Linear constraints converge in one Newton step** (closed-form `X' = X − Jᵀ(JJᵀ)⁻¹r`); **nonlinear constraints** iterate the same step until `‖r‖ ≤ tolerance` or `max_iter`; pathological starting points (e.g. `X=0` with `g(X)=‖X‖²−4` → singular Jacobian via `2X=0`) log WARNING + return best-so-far rather than raise or loop forever.  **Local-mode mask projects only the masked subset**: `J_local = J_full[:, mask]`, then `X[mask] -= J_localᵀ(J_local J_localᵀ)⁻¹ r`; values outside the mask are touched by no code path.  Pinned by a byte-equality test.  **All-False mask path**: warns + returns X unchanged (rather than raising) — the user explicitly chose a mask that cannot fix the residual; `project` is a primitive whose role is to compute, not police caller intent.  **`np.linalg.lstsq` on `J·Jᵀ`** rather than `np.linalg.solve` for rank-deficient robustness — same pattern as OP-032's NNR projection.  **Default-mode-per-domain table** is case-insensitive: hydrology/remote-sensing → local, seismo-geodesy/geodesy → coupled, unknown/None → naive.  **No audit emission** — `project` is a primitive called from OP-032 / OP-050 / cf_coordinator, all of which emit their own audit at the orchestration layer; double-emission would be wrong.  30 tests in `test_compensation_op051.py`; full backend at 1530 pass; ruff clean.  Code-reviewer approved with no blocking issues; flagged two non-blocking follow-up cleanup tickets (the `try/except ImportError` fallback at `cf_coordinator.py:243–251` is now dead code on every supported install; threading iteration count + final residual back to callers via a return-tuple instead of just-logging would let OP-032/OP-050 record `convergence_iters` in their audit).

## OP-033 aggregate (Tier-3) read-only summary metrics

Created `backend/app/services/operations/tier3/aggregate.py` — the third Tier-3 op, alongside OP-030 decompose and OP-032 enforce_conservation in the same package.  Public surface: `aggregate(X, segments, metric, aux=None) → dict[segment_id, metric_value]`, `METRIC_REGISTRY` + `@register_metric` decorator, and 10 built-in metrics (`peak`, `trough`, `duration`, `area`, `amplitude`, `period`, `tau`, `bfi`, `sos_eos`, `m0`).  **Signature deviation from the ticket pseudocode** (`aggregate(segments, metric, aux)` with `s.X` per segment): explicit `X` is the right call because adding a per-segment array to `DecomposedSegment` would duplicate storage and break the OP-030 design intent of "bounds-only segment, blob optional".  This matches `enforce_conservation`'s existing contract of taking `X` separately.  **Reuses `DecomposedSegment` from OP-030** so a single `decompose(...)` call produces segments that can be `aggregate(...)`-d immediately.  **Read-only contract**: frozen `DecomposedSegment` (already from OP-030); `np.asarray(X, dtype=np.float64).ravel()` produces an `arr` that is only sliced and read; per-metric callables only read.  `DecompositionBlob` is mutable (not frozen) but no code path writes to it — pinned by `test_aggregate_does_not_mutate_decomposition_blob` which deepcopies the blob before the call and checks every component / coefficient / fit_metadata key / residual byte-for-byte after.  **Crash safety**: each metric callable runs inside `try/except Exception` (with `noqa: BLE001`) — a single buggy domain-pack metric never crashes the whole table; the failing segment gets `None` and a warning is logged.  Tested via a deliberately-raising metric.  **`None` semantics** for "not applicable": `period` on plateau without decomposition, `tau` on a non-GrAtSiD blob, `bfi` when `Σ Q ≤ 0` or no baseflow component, `sos_eos` when amplitude is zero (flat segment), `m0` when any of `shear_modulus` / `fault_area` / `slip_from_segment` is missing from `aux`.  **`sos_eos`** returns a dict `{sos: int, eos: int, threshold_value: float}` so the UI table can render the threshold alongside the indices; default `threshold_percent=20` per Jönsson 2004 Table 1.  **`m0`** requires `aux` (rock properties not derivable from the segment alone) — Aki & Richards 2002 Ch. 3 `M₀ = μ · A · s`.  **No audit emission** — read-only op; consistent with how the codebase treats other read-only paths (audit is a coordinator-level concern).  39 tests in `test_aggregate_tier3.py`; full backend at 1500 pass; ruff clean.  Code-reviewer approved with no blocking issues; flagged three follow-up enhancements as non-blocking (optional `strict=True` to re-raise per-metric failures instead of returning `None`; optional `audit_log` kwarg if reproducibility-of-dashboard-queries becomes a requirement; per-function paper citations on the four "trivial stat" metrics — none of these are needed for OP-033 to ship).

## OP-032 enforce_conservation (Tier-3) with 4 laws

Created `backend/app/services/operations/tier3/enforce_conservation.py` plus a wired-up `tier3/__init__.py`.  Public surface: `enforce_conservation(X_all, law, compensation_mode='local', aux=None, *, tolerance=None, event_bus=None, audit_log=None) → (X_edit, ConservationResult)`, frozen `ConservationResult` and `ConservationAudit` dataclasses, `LAW_REGISTRY` + `@register_law` decorator (extensible at import time so external packs can register more laws), `UnknownLaw` exception, `HARD_LAWS` / `SOFT_LAWS` frozensets, `DEFAULT_TOLERANCE` per-law dict.  **Four laws ship in MVP**: `water_balance` (P − ET − Q − ΔS = 0; soft, tol 1e-6; Eckhardt 2005), `moment_balance` (trace(M) = 0; soft, tol 1e-9; Aki & Richards 2002 Ch. 3), `phase_closure` (φ12 + φ23 − φ13 ≡ 0 mod 2π; hard, tol 0.1 rad; De Zan 2015), `nnr_frame` (Σ rᵢ × vᵢ = 0; hard, tol 1e-9; Altamimi 2011).  **Three compensation modes per law**: `naive` (no projection, just reports residual), `local` (entire residual absorbed by one preferred signal — ΔS for water, Mzz for moment, φ13 for phase), `coupled` (least-squares Lagrange distribution — water gives ±r/4 across all four; phase gives ∓r/3 across the three).  **OP-051 deferral note**: the ticket says "delegates to OP-051" but OP-051 isn't shipped; per-law projection math IS the math, so delegating would just bounce through OP-051 to the same per-law branches.  Module docstring's "OP-051 contract" section documents the migration path so a future reader sees the deferred refactor.  Same pattern cf_coordinator already uses for OP-051 import fallback.  **Hard vs soft semantics**: hard-law non-convergence logs WARNING + sets `converged=False`; soft-law logs INFO + same flag.  **Neither raises** — UI-010's residual-budget bar needs the residual numeric to render even (especially) when not converged.  Tests `test_hard_law_non_convergent_returns_converged_false_without_raising` and `test_soft_law_non_convergent_does_not_raise` lock this in.  **Phase wrapping gotcha**: `_wrap_phase` wraps closure into (−π, π] BEFORE projection so that a closure of ~2π (which is physically zero modulo 2π) doesn't inflate the residual; tested via `test_phase_closure_wraps_2pi_ambiguity`.  **NNR projection math** (Altamimi 2011 Eq. 4–6): `A ω = b` where `A = Σ (‖rᵢ‖² I − rᵢ rᵢᵀ)`, `b = Σ rᵢ × vᵢ`; `np.linalg.solve` with `lstsq` fallback on rank-deficient `A` (≥3 non-collinear stations make it invertible — the 3-station check is necessary but not sufficient; collinear stations exercise the lstsq path).  **Naive-mode aliasing**: `dict(X_all)` is a shallow copy so `X_edit[key]` shares ndarray refs with the input in naive mode (acceptable per the documented contract — input is not mutated, but output may share refs).  Local/coupled modes always produce fresh arrays.  35 tests in `test_enforce_conservation_tier3.py`; 1434 backend pass; ruff clean.  Code-reviewer approved with no blocking issues; flagged the mutable-`extra: dict` field on `ConservationResult` as a frozen-dataclass nit (mutable container inside a frozen dataclass — the strict CLAUDE.md reading would prefer a tuple-of-pairs, but the field is unused so far).

## OP-030 decompose (Tier-3)

Created `backend/app/services/operations/tier3/decompose.py` (the `tier3/__init__.py` was empty before this ticket).  Public surface: frozen `DecomposedSegment` (segment_id / start_index / end_index / label / scope / decomposition), frozen `DecomposeAudit` (op_name / tier / segment_ids / methods_used / domain_hint / refit_reason / extra), `REFIT_REASON = 'user_tier3_decompose'`, and `decompose(X, segments, domain_hint=None, *, event_bus=None, audit_log=None) → list[DecomposedSegment]`.  The function is purely functional at the segment-list level: input segments are frozen, the return is a *new* list built via `dataclasses.replace(seg, decomposition=blob)`.  **Domain-hint priority** is function-arg > `seg.scope['domain_hint']` > `None` (generic fitter); resolved per-segment so a single call can mix hints.  **Critical mutation gotcha**: the fitter may return a blob from an internal cache (some fitters memoise sub-results), so `_annotate_refit` does `copy.deepcopy(blob)` *before* stamping `fit_metadata['refit_reason']` and `fit_metadata['domain_hint']` — without the deep-copy, the same cached blob would accumulate metadata across calls.  Audit emission goes via the OP-041 `EventBus` and `AuditLog` (`default_event_bus.publish('decompose', audit)` + `default_audit_log.append(audit)`), with kwarg overrides for test isolation.  No `LabelChip` is emitted because `decompose` does not change a segment's *shape label* — only its decomposition; this is intentional.  **AC interpretation note**: "validator from SEG-019 invoked" — SEG-019's `dispatch_fitter` doesn't expose a separate blob validator; the `KeyError` on an unknown shape label IS the validation path.  If a granular blob validator is added later, `decompose` will need an extra call site after the fitter returns.  Bounds validation rejects `start_index < 0`, `end_index >= n`, `end_index < start_index` with descriptive `ValueError` messages naming the offending `segment_id`.  Empty segment list is allowed and produces an audit entry with empty `segment_ids` / `methods_used` tuples.  27 tests in `test_decompose_tier3.py`; 1426 backend pass (the same 2 pre-existing unrelated failures untouched).  Tests use isolated `EventBus` / `AuditLog` instances except for one explicit "uses default log" test that pops the just-appended entry from `default_audit_log._records` in a `try/finally` (less invasive than `clear()`, which would wipe other tests' records — `noqa: SLF001` annotated).

## SEG-023 Remote-sensing semantic pack (phenology, disturbance, InSAR)

Third semantic pack on top of SEG-021's framework: 14 labels — `greenup, senescence, peak_of_season, dormancy, disturbance, recovery, wet_up, dry_down, heatwave, coldwave, cloud_gap, APS, seasonal_deformation, linear_deformation`.  `__init__.py` gets a one-line `from . import detectors_remote_sensing` (alphabetically between hydrology and seismo-geodesy).  **Delegation pattern continues**: `bfast_breakpoint` lazily imports `fit_bfast` (reads `coefficients["breakpoint"]`, `level_left`, `level_right`); `landtrendr_positive_slope_post_disturbance` lazily imports `fit_landtrendr` (reads `coefficients["slope_2"]` — the legacy 2-segment schema; falls back to `slopes[-1]` for future generalised LandTrendr or to OLS slope when the fitter is unavailable); `etm_linear_rate` lazily imports `fit_etm` (reads `coefficients["linear_rate"]`).  The `seasonal_deformation` AC mentions "MSTL annual component" but the implementation uses a cheap FFT dominant-period check rather than fitting MSTL per segment — fitting MSTL would re-decompose what the parent shape classifier already established; the 365 ± 30 day band is enforced via the YAML `is_annual_period` predicate.  **Critical Jönsson 2004 §2 nuance**: SOS/EOS thresholds are defined relative to the *full season's* min-to-max range, not the segment's local range — using a segment-local range would mean every monotonic segment trivially crosses any threshold (a rising linspace from 0.1 to 0.6 has segment_min=0.1, segment_max=0.6, so a 20% threshold = 0.2 sits inside the segment by construction; the test would always pass).  The `_annual_min` / `_annual_max` / `_annual_amplitude` helpers therefore *prefer* caller-supplied values from context (`annual_min`, `annual_max`) over segment-local stats — documented in `_annual_amplitude`'s docstring.  Tests pass `annual_min=0.1, annual_max=0.6` in context to exercise the full-season pathway.  **Recovery test slope gotcha**: my first version used `np.linspace(0.2, 0.6, 50)` — slope = 0.4/49 ≈ 0.008, BELOW the default `min_recovery_slope=0.05`.  Fixed by using `np.linspace(0.2, 1.2, 10)` so slope ≈ 0.111 > 0.05; the tests now distinguish "passes recovery threshold" vs "below recovery threshold" cleanly.  **Same callable serves multiple labels** continues from SEG-021/022: `timesat_threshold` for greenup+senescence (predicate disambiguates by sign), `slope_plus_context` for wet_up+dry_down, `percentile_threshold_plus_duration` for heatwave+coldwave.  **`cloud_gap` is metadata-only**: reads `context['is_cloud_gap']` from the data loader's missingness mask, NEVER touches `X_seg` values — explicit test `test_cloud_gap_does_not_call_interpolation` passes a NaN-filled segment and verifies no NaN propagation.  62 tests in `test_semantic_pack_remote_sensing.py`; full backend suite at 1597 pass; ruff clean.  The cross-cutting parametrised "every detector returns (bool, float)" test now iterates the full 34-detector registry (9 hydrology + 14 seismo-geodesy + 11 remote-sensing).

## SEG-022 Seismo-geodesy semantic pack

Added the second semantic pack on top of SEG-021's framework: 14 labels — `P_arrival, S_arrival, coda, surface_waves, tremor, coseismic_offset, postseismic_relaxation, interseismic_loading, SSE, seasonal_signal, common_mode_error, tropospheric_delay, unwrapping_error, antenna_offset`.  The new `detectors_seismo_geodesy.py` is auto-imported by `__init__.py` (one-line add) so the registry is populated before `load_pack('seismo_geodesy')` is called.  **YAML filename / pack-name divergence**: file is `seismo_geodesy.yaml` (Python identifier — `load_pack('seismo_geodesy')`) but the *internal* pack `name` is `seismo-geodesy` (hyphenated, matches the ticket schema).  **Detector delegation**: `etm_step_from_known_origin` and `fit_log_or_exp` import `fit_etm` lazily to delegate parametric fitting to SEG-013 (no duplicated step / log-exp logic).  `gratsid_bump` (SSE) is a heuristic — duration in [7, 365] days at `samples_per_day` × smooth-onset/decay via diff-jitter — *not* a literal `fit_gratsid` call, because invoking GrAtSiD per-segment would re-fit a feature list that the parent shape classifier already established; the SSE candidate's metadata is stored in the form OP-025 expects.  **STA/LTA implementation gotcha**: my first sanity test asserted `ratio[150] > ratio[50]` for an onset at index 100 — failed because by index 150 the LTA window has fully absorbed the post-onset amplitude and the ratio drops back to ~1.  STA/LTA peaks AT the onset, not far past it — the corrected test compares `ratio[100:110].max()` against the pre-onset baseline `ratio[60:90].mean()`.  This is the canonical Allen-1978 picker behaviour: it triggers ON the onset, then the LTA catches up and the ratio relaxes.  **Context-mutation gotcha**: my first unwrap-error test asserted `ctx['snap_2pi_multiple_count'] == 1` after calling `match_semantic_label(...)` — failed because SEG-021's matcher *copies* the user's context dict before passing it to the detector (so detector mutations don't leak between labels in `label_segment`).  Fixed by calling the detector directly when verifying context mutation, and only using `match_semantic_label` to verify the `(matched, confidence)` tuple.  Worth remembering for any future test that wants to inspect detector-augmented context: go through `DETECTOR_REGISTRY[name]` directly, not the matcher.  **`phase_jump_2pi_detector` k=0 carve-out**: snap-to-2π would return `k=0, within_tol=True` for any flat segment (zero step magnitude is trivially within 0.1π of 0·2π), so the detector explicitly rejects `k=0` to avoid tagging every flat segment as an unwrap error — covered by `test_unwrapping_error_zero_step_rejected_even_if_within_tolerance`.  **`common_mode_pca` multi-station guard**: returns `(False, 0.0)` and sets `not_applicable=True` when the caller did not supply `station_residuals` or supplied fewer than 2 rows — explicitly required by the AC.  **`obspy` is NOT a dependency**: docstring promises an optional fallback, but the module never imports obspy; scipy's `hilbert` is used (already in requirements) with absolute-value fallback if scipy is unavailable.  Test `test_sta_lta_works_without_obspy` confirms.  67 tests in `test_semantic_pack_seismo_geodesy.py`; 1524 backend pass (the same 2 pre-existing unrelated failures untouched).  The cross-cutting parametrised "every detector returns (bool, float)" test in `test_semantic_packs.py` now iterates over the full 23-detector registry (9 hydrology + 14 seismo-geodesy).

## SEG-021 Hydrology semantic pack (baseflow, storm, ENSO, PDO)

Created `backend/app/services/semantic_packs/` with `core.py` (frozen `SemanticPack`/`SemanticLabel` dataclasses, `DETECTOR_REGISTRY` populated by `@register_detector` decorator, AST-validated `evaluate_predicate`, YAML `load_pack`, `match_semantic_label`, `label_segment`), `detectors_hydrology.py` (9 detectors), and `hydrology.yaml` (9 labels — `baseflow / stormflow / peak_flow / rising_limb / recession_limb / snowmelt_freshet / drought / ENSO_phase / PDO_phase`).  Each label is a `(shape_primitive, detector, context_predicate)` triple per *Formal Definitions* §2.3.  **Detector contract**: `(X_seg, shape_label, context) → (matched: bool, confidence: float)`; detectors *mutate* the `context` dict in place to add the metrics that the matching predicate references (e.g. `Q_mean`, `peak_Q`, `slope`, `dominant_period_years`).  The matcher then evaluates the YAML predicate against the augmented context.  The `baseflow` detector delegates to the SEG-016 Eckhardt fitter (`fit_eckhardt`) rather than re-implementing the recursive filter.  `slope_sign_plus_context` is the *same* detector for `rising_limb` and `recession_limb` — the predicate distinguishes by sign-of-slope and neighbour flags (`preceded_by_baseflow`, `follows_peak_flow`); the caller is expected to populate those flags from segment-context (since one segment cannot infer them from `X_seg` alone).  **Predicate evaluator security**: `evaluate_predicate` uses Python `eval` on a restricted namespace (`__builtins__: {}` + a fixed allow-list of `abs / max / min / len / all / any / round / sum`) gated by an AST whitelist (BoolOp, Compare, BinOp, UnaryOp, Constant, Name, Call, List, Tuple — explicitly excludes Attribute, Subscript, Lambda, Comprehensions, IfExp, Set/Dict literals, Starred, walrus, `__import__`). ## SEG-017 LandTrendr piecewise-linear fitter (EO trajectories)

Replaced the LandTrendr stub at `backend/app/services/decomposition/fitters/landtrendr.py` with a full Kennedy 2010 implementation.  Pipeline: (1) `find_candidate_vertices` — start from endpoints, iteratively add the time index whose residual against the *current* anchored fit is largest (greedy, Kennedy 2010 §2.2); (2) `_prune_iteratively` — drop the interior vertex whose removal least increases SSE, recording every (k, vx, sse) snapshot; (3) penalised pick — among truncations with k ≤ `max_vertices`, choose minimal `SSE + penalty_per_vertex · k`.  **Critical algorithmic gotcha**: an unanchored joint OLS over all vertex Y values puts the largest residual at the endpoints (the OLS line is mean-centred — boundary samples are far from it), so the candidate generator picks endpoint-NEIGHBOUR samples instead of true interior knots — got `[0, 5, 6, 239]` instead of `[0, 100, 200, 239]` on a clean 3-knot test fixture. ## SEG-015 BFAST decomposition fitter (EO breakpoints)

Replaced the placeholder `backend/app/services/decomposition/fitters/bfast.py` with a full BFAST fitter (Verbesselt 2010 classical + Masiliūnas 2021 lite). ## OP-026 Noise Tier-2 ops (5 ops)
